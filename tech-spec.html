切分策略

不按文件切分，按语义段落切分（每段 300~500 token）。切分边界优先在标题、空行、句号处断开。每个向量块保留元数据：文档标题、来源、创建时间、作者、所属项目、在原文档中的顺序位置（用于检索后按顺序拼接上下文）。

异常预警机制

三层预警逻辑：

预警等级触发条件响应动作
紧急 核心指标骤变（如充值成功率 < 50%、DAU 暴跌 > 30%） 立即推送至运营群，AI 给出初步归因
警告 指标偏离基准 10~20%，连续 2~3 天 每日报告中突出显示，推荐排查方向
观察 指标缓慢偏离，趋势性变化 周度报告归类，长期跟踪

技术实现

数据接入：通过数数科技 Open API 拉取原始事件数据，定时任务（每 15 分钟）增量同步。后端日志通过专门接口接入，弥补数数科技未覆盖的业务侧数据。

指标计算：原始数据进 PostgreSQL，预聚合层计算各指标日/小时粒度结果，存入 metrics 表（按模块、指标名、时间维度建索引）。预警规则配置存 alert_rules 表，定时扫描触发。

看板展示：Grafana 或自建前端展示，按模块分页，支持下钻查看明细。预警通过 Lark 机器人推送到运营群。

归因分析：异常触发时，调用 Claude API，传入异常指标 + 同期相关指标 + 业务上下文，生成初步归因建议。这部分依赖版本一的认知底座（产品文档、规律库）。

数据表结构（关键表）

metrics（指标值表）
- id, module, metric_name, value, granularity, ts
- 索引：(module, metric_name, ts), (granularity, ts)

alert_rules（预警规则表）
- id, module, metric_name, condition, threshold, level, enabled
- 例：{module:'收益健康', metric:'充值成功率', condition:'< 75% for 3 days', level:'紧急'}

alert_history（预警历史表）
- id, rule_id, triggered_at, metric_value, ai_attribution, status
- ai_attribution 字段存 AI 给出的归因结论 JSON

人天估算依据

数数科技 API 接入 + 数据同步 2~3 天，六模块指标定义 + 计算逻辑 2~3 天，预警规则系统 1~2 天，看板配置（Grafana 或自建）2~3 天，归因分析 AI 集成 1~2 天。合计 8~12 天。自研方案需自建数据采集 + 计算引擎，增加 7~8 天。

待评估

分群策略

不做单一维度分群（如"所有大 R"），而是用三层叠加分群——例如"渠道 A 来的偏好 Teen Patti 的中 R 用户"。这样的分群更精准，营销策略对应也更明确。

分群示例典型特征对应策略方向
新注册大 R 注册 7 天内首充 ≥ 500₹，下注规模快速上升 VIP 客服快速接入、专属充值奖励
休眠中 R 历史中 R，最近 30 天下注下降 70%+ 定向召回活动、好友邀请奖励
套利风险用户 注册即领奖，下注集中在低风险游戏，提现频繁 风控标记、活动门槛限制
偏好转移期用户 原本主玩 A 游戏，最近开始尝试 B 游戏 B 游戏的引导奖励、组合推荐

虚拟用户生成（用于策略预判）

画像不只是分析工具，也是策略预判的输入。系统会基于真实用户分布生成虚拟用户群——保持各分群的占比和典型特征，但不绑定真实身份。新策略上线前先在虚拟用户群里跑模拟，预测各分群的反应。

生成方式：从真实数据中抽取分群核心特征（如"中 R 用户的平均下注、留存曲线、付费时间分布"），生成符合分布的合成用户。规模可配置，常用 1 万~10 万的样本量做策略预判。

数据表结构

user_profiles（用户画像主表）
- user_id, behavior_features JSONB, source_features JSONB, preference_features JSONB
- updated_at, version
- 索引：(user_id), GIN 索引 on JSONB 字段

user_segments（分群定义表）
- segment_id, name, rule_definition JSONB, user_count, last_calc_at

user_segment_members（分群成员表）
- segment_id, user_id, joined_at, left_at
- 用于跟踪用户在分群间的迁移

virtual_users（虚拟用户表）
- vuser_id, segment_id, simulated_features JSONB, batch_id
- batch_id 标识同一次生成的批次，便于策略预判时复用

技术实现

画像计算：基于版本一的数据底座，每日定时任务计算所有用户的三层特征。增量更新（只算有行为变化的用户）。

分群引擎：分群规则用 JSON 描述，存数据库可配置。引擎遍历用户画像匹配规则，生成分群成员关系。规则变更时支持回算历史。

偏好层推导：用统计方法（不是机器学习黑盒），例如"过去 30 天 70% 下注集中在 Teen Patti"→ 偏好 Teen Patti。可解释性强，便于运营理解。

虚拟用户生成：用 Python 脚本，按分群从真实数据抽样特征分布，生成合成用户存入 virtual_users 表。

人天估算依据

行为层 + 来源层特征计算 3~4 天，偏好层推导逻辑 2~3 天，分群引擎 + 规则系统 2~3 天，虚拟用户生成 2~3 天，画像更新调度 + 测试 1~2 天。合计 10~15 天。自研方案需自建复杂的特征工程平台，增加 10 天。

待评估

分群粒度需要平衡——过粗（如只分大中小 R）信息量不足；过细（如几十个交叉分群）样本量不够导致策略预判失真。建议初期 8~12 个核心分群，后续根据使用情况调整。

T2.3 策略预判引擎（数字孪生机制） 后端 · 15~20 天

核心目标

在新运营策略真正上线前，用虚拟用户群在虚拟环境中模拟一遍，预测对各分群用户的影响、整体数据变化、潜在风险。让决策从"上线后才知道效果"变成"上线前已有合理预期"。

推演方法

不做大规模 ML 模型训练。这种规模的训练在我们的数据量级和团队配置下不现实，且可解释性差。

采用规律匹配 + AI 推理的混合方法：
第一步：解析策略文档（规则、奖励、目标人群），结构化为可推演输入。
第二步：在规律知识库（来自版本一的历史复盘 + 数数科技的指标基准）中匹配类似策略的历史效果。
第三步：基于匹配到的历史规律，结合当前用户分布，推演各分群对新策略的响应概率。
第四步：Claude API 综合分析，生成自然语言预测报告，附置信度。

置信度分级

置信度触发条件使用建议
高 历史有 100+ 次类似策略 + 用户分布匹配度高 结论可作为决策依据
中 历史 20~100 次类似 / 有部分新变量 建议小规模灰度验证后推广
低 全新类型策略 / 历史数据稀少 明确说"数据不足"，建议先做 A/B 测试

校准机制

策略真实上线后，系统自动记录"预测值 vs 实际值"的差异，存入 prediction_calibration 表。每月生成校准报告——哪类策略预测准确率高、哪类容易偏差、偏差方向是什么。基于校准结果迭代规律库，让推演越来越准。

厂商游戏作为状态变量：不仿真游戏内部数值（RTP、赔率厂商锁定），但将玩家当前游戏场景（在玩什么、最近输赢状态、游戏切换行为）作为推演变量。同一充值活动，玩家刚连输 vs 刚赢了一把，响应率完全不同。

数据表结构

strategy_simulations（策略仿真表）
- sim_id, strategy_doc_id, status, started_at, finished_at
- input_params JSONB, result JSONB, confidence_level

simulation_segment_results（分群结果表）
- sim_id, segment_id, predicted_response_rate, predicted_revenue_delta
- 用于查看每个分群的具体预测

prediction_calibration（预测校准表）
- prediction_id, predicted_value, actual_value, deviation
- strategy_type, recorded_at
- 用于持续校准准确率

技术实现

策略文档解析：Claude API 解析策略文档，提取规则、目标用户、奖励结构、时间窗口等结构化字段。

规律匹配：用向量检索（pgvector）从历史规律库找类似案例，结合规则引擎过滤适用条件。

推演计算：Python 服务，遍历各分群虚拟用户，应用规律得出响应概率，汇总成报告。

报告生成：Claude API 把数值结果转换为自然语言报告，包含"建议推进/调整/不推进"的明确结论。

人天估算依据

策略文档解析逻辑 2~3 天，规律匹配引擎 3~4 天，推演计算服务 4~5 天，置信度模型 1~2 天，校准机制 + 报告生成 2~3 天，前端展示 + 联调 3 天。合计 15~20 天。自研方案需自建特征工程和评分模型，增加 15 天。

待评估

第一版仿真误差预计在 15~30%，需要至少 2~3 个版本节点的校准数据才能稳定到 10% 内。冷启动期间应明确告诉运营团队"这是参考、不是承诺"，避免初期失望。

关键设计原则

① 永远附数据依据。每个建议都列出"基于什么历史案例""数据支撑是什么"，让使用者能验证而不是盲信。

② 永远标置信度。从"高/中/低"明确告知判断的可靠性。低置信度场景明说"数据不足，建议小范围验证"。

③ 永远提风险。不只说方案好处，主动列出潜在风险点和需要监控的指标。

④ 价值观过滤。调用思想层（来自版本一）做最终审核——确保建议符合公司责任博彩边界等价值观要求，违反则不输出该建议。

数据表结构

- session_id, user_id, mode, input JSONB, status

inference_results（推演结果表）
- result_id, session_id, recommendation TEXT, evidence JSONB
- confidence_level, risks JSONB, generated_at

inference_feedback（反馈表）
- feedback_id, result_id, was_adopted, actual_outcome
- 用于校准 AI 推演的准确率

人天估算依据

三种模式的 Prompt 设计 + 调试 3~4 天（这部分是核心工作量），上下文检索逻辑 2~3 天，与策略预判引擎对接 2~3 天，价值观过滤层 1~2 天，反馈记录 + 校准接口 1~2 天，测试调优 1 天。合计 10~15 天。

待评估

Claude API 调用成本是版本二的主要工具成本。每次推演调用 1~3 次 API（取决于复杂度），单次成本约 ¥0.3~1.5。建议立项后先做用量测算，避免上线后才发现成本超预期。

指标看板页

六模块指标卡片式展示，支持时间范围切换、分群下钻、对比模式（与上周/上月/同期对比）。卡片右上角显示当前预警状态。点击卡片进入明细页，展示趋势图 + AI 自动归因。

策略推演页

三种模式各自有专属入口和表单：

目标导向：填写目标（"提升中 R 留存 5%"）+ 时间范围 → 系统给候选策略列表，每个附预期效果。
方案预判：上传或粘贴方案文档 → 系统输出预期效果 + 风险 + 建议调整，分群拆解。
指标预测：选择要变更的参数（如奖励金额）+ 变化幅度 → 系统输出受影响指标的预测值。

所有推演结果支持导出为 PDF 报告，便于内部分享。

历史报告页

所有推演结果按时间归档，支持搜索、筛选、对比。每条报告显示"是否被采纳"和"实际结果 vs 预测"——这是 AI 准确率的可视化记录，让团队能直观看到系统在变得更准。

技术选型

建议方案：React + Tailwind CSS + Recharts（图表库）+ shadcn/ui（组件库）。前端独立部署，通过 REST API 与后端通信。

不推荐 Retool 的原因：版本二的界面交互比版本一复杂得多——文档上传、分群下钻、对比图表等。Retool 在这个复杂度下灵活性不足，自建反而更划算。

响应式：主要适配桌面端，移动端简化版只展示指标看板和预警，不支持推演操作。

人天估算依据

指标看板页 3~4 天，策略推演页（三种模式）4~5 天，历史报告页 2~3 天，预警中心 1~2 天，整体路由 + 状态管理 + 联调 1~2 天。合计 10~15 天。

待评估

界面的使用人数和频率会影响后端 QPS 设计。建议先评估有多少运营人员会用、日均使用频次，确定服务器规格。

版本二整体架构小结：数数科技 + 后端日志 → 指标计算层（六模块）→ 监控应用层（看板 + 预警 + 归因）。同时用户画像三层叠加 → 虚拟用户群 → 策略预判引擎，对接 AI 推演引擎完成"决策辅助"闭环。运营通过专属界面输入策略，看到预测报告。版本二在版本一认知底座的基础上，把"凭经验"升级为"基于数据 + 模拟"。
