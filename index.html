执行摘要

打造一个与公司共同成长的 AI 决策伙伴，分四个版本递进建设：

版本一 建立公司认知底座（组织/工作/产品/数据/价值观）
版本二 决策辅助 + 策略预判（数字孪生机制）
版本三 行业视野与竞品洞察（对标分析 + 差异化机会识别）
版本四 群体智慧积累与赋能（经验沉淀 + 文档代码审查）

本次立项建议：先批版本一。版本一是所有后续版本的基础，单独来看就能交付可验证的价值——让系统完整掌握公司的组织、工作、产品、数据和价值观。后续版本根据版本一的实际运行效果再分别决策。

关键路径：版本一是所有后续版本的前置依赖。版本二、三可并行，版本四依赖前三版本稳定运行 3~6 个月后启动。
思想层并行推进：价值观与行为准则的定义与版本一开发并行，不互相阻塞。数据底座的工作可以先启动，思想层 1.0 完成后再对接过滤层。
人天与工具成本：详见第 05 章「资源与成本」。基于现有后端 × 1、前端 × 1 团队配置估算，提供最小方案和自研方案两套选项。

需要管理层决策：批准版本一立项，约定思想层 1.0 的输入节奏（不作为启动前置，但需要在版本一运行中逐步完成）。

01

我们要做什么

我们正在构建一个 持续学习、持续生长的 AI 决策系统。它不是一个固定功能的工具，而是一个会随着公司的每一次决策、每一个项目、每一位员工的积累而不断进化的伙伴。

当我们需要上线一个新的运营活动，它能在上线前告诉我们预期效果、潜在风险、以及竞品在类似阶段是怎么做的。当我们面临方案选择，它能调取公司历史上所有相关决策的经验，给出有数据支撑、有案例论证的建议，而不是主观判断。

现在我们面临的核心问题是：决策依赖个人经验，经验无法沉淀传承；方案上线后才知道效果好不好；团队的集体智慧分散在每个人的头脑里，没有办法被系统性地调用。

这个系统要解决的，就是把分散的经验变成可调用的集体智慧，把事后复盘变成事前预判。

决策前

主动提供方案

基于市场竞品、跨行业案例、内部历史数据，给出有理由和数据支撑的思路与落地建议

决策中

查漏补缺

对方案进行全面审查，识别遗漏和风险，用群体智慧覆盖个人盲点

决策后

持续校准

记录每次决策结果，对照预测持续修正，越用越准，越用越懂我们

02

系统终态与总目标

系统完全建成后，它在公司里的角色是：

一个与公司共同成长的决策伙伴。它了解我们的业务现状和数据，理解我们的思维方式和判断标准，能从行业竞品和跨类型产品中总结归纳核心思路，为我们的需求提供有合理理由、有数据论证的落地方案。同时，它积累了公司所有员工的正向经验，能协助每一个人审查文档、检查代码逻辑、补充上线验收流程，用集体智慧提升每个个体的工作水平。

达到终态需要四个版本的递进建设。每个版本都有独立的交付价值，不依赖后续版本才能发挥作用。

版本一基础

公司认知底座

系统理解我们是谁、怎么思考、有什么数据。这是后续所有能力的地基。

版本二核心

决策辅助 + 策略预判

监控业务指标，模拟用户和游戏环境，对新策略做上线前的效果预判。

版本三视野

行业洞察

对标竞品，分析他们在不同阶段的决策逻辑，为我们提供有依据的参考。

版本四 · 智慧积累

群体智慧沉淀，赋能每一个人

在前三个版本运转过程中持续积累经验，形成文档规范、开发流程标准、审查能力。系统能协助每个员工检查方案、审查代码、补充流程，用集体智慧提升个体水平。

竞品版本追踪 决策逻辑推断 跨行业案例库 差异化机会识别

在前三个版本运转的过程中，系统持续积累正向经验，沉淀为职业智慧知识库。积累了足够的文档规范、功能开发流程、决策案例后，系统能协助每个员工审查方案文档、检查代码逻辑、补充上线验收流程，用公司集体智慧提升每一个个体的工作水平。

职业智慧知识库 文档审查能力 代码逻辑检查 群体审查机制 员工成长提醒

各版本模块细节

以下对每个版本的核心模块进行框架设计说明，包含实现原理、关键难点，以及目前存在分歧、需要进一步探讨的问题。

版本一模块：建立公司认知底座

版本一验收标准

完成以下 5 项即视为版本一达标，具备进入版本二的基础条件：

① 数据源接入完成：Lark 工作文档 + 禅道任务记录 + 代码仓库 + 产品功能文档四类数据源完成接入并稳定同步，每日新增数据量可监控。
② 历史文档入库达标：产品功能文档 100% 入向量知识库，近 12 个月的复盘文档、版本发布记录、主要方案文档入库完成。
③ 系统能回答基础问题：能准确回答"我们有哪些产品、各自有什么功能"、"上个季度做了什么项目"、"某个功能为什么这样设计"这类基于公司事实的问题。
④ 生成认知文档初稿：系统输出至少 5 份思维方式初稿文档（面对需求不明确的处理方式、决策收敛模式、延期项目的规律等），经管理层审阅后至少 3 份确认入库。
⑤ 隐私过滤有效性：首月目标——建立基线，误拦率 < 10%、漏拦率 < 5%，完成规则迭代；稳定期目标（3 个月后）——误拦率 < 5%、漏拦率 < 2%。统计口径：每周人工抽样 200 条过滤日志（含通过/拦截/待审核三类），按分类结果复核计算。对员工反馈的隐私问题能及时响应和调整。

立项后 30 天交付物

30 天内完成以下 5 项，让老板和团队能看到实际进展，不是空口承诺：

① 数据源权限全部申请完成：Lark 企业自建应用审批通过、禅道 API token 开通、代码仓库访问权限配置、数数科技 Open API 接入。
② 首批历史文档入库：产品功能文档 + 近 3 个月复盘文档完成向量化入库，知识库最小可用。
③ 知识库查询 Demo：后台可以通过自然语言查询已入库内容，返回相关段落。管理层可现场体验。
④ 隐私过滤规则 1.0：三层过滤架构上线，首版关键词规则完成，开始收集过滤日志。
⑤ V1 管理后台初版：数据源状态看板 + 知识库浏览界面可用，支持观察同步情况。

1.0 思想层 — 价值观与行为准则 与版本一并行 版本一

框架设计

思想层是整个系统的价值观地基，决定 AI 用什么标准去判断和建议。分为两个层次：行业层（公司使命愿景价值观、责任博彩边界、印度市场合规边界）和社会层（更底层的判断逻辑框架）。这些内容以结构化文档的形式写入系统，作为所有推演的基础约束。

实现原理

由公司管理层主动定义，以自然语言文档形式输入。系统在每次给出建议时，会先用思想层内容过滤——确保建议符合公司价值观和合规边界，再输出结论。

推进节奏

思想层的定义与版本一开发并行推进，不互相阻塞。数据底座、文档接入、能力地图等工作可以先启动。思想层 1.0 完成后再对接过滤层即可——先有粗略的 1.0 版本让系统启动，后续迭代细化，避免追求完美导致无限拖延。需要外部输入的内容：公司使命愿景价值观、责任博彩的思考路径和红线、社会层判断逻辑框架。

1.1 组织架构与团队能力地图 版本一

框架设计

建立公司组织的完整认知：整体架构图谱、各部门职责划分、人员构成、汇报关系、各角色的核心工作内容和产出标准。这是系统理解"公司里有哪些人、分别负责什么"的基础。

团队能力地图

基于日报周报、代码提交记录、功能完成情况，了解团队在不同领域的能力分布——哪些方向人手充足、哪些方向是瓶颈、跨职能协作的效率如何。这份地图帮助系统在给建议时知道"什么方案能落地、什么方案超出团队能力边界"，随数据积累持续修正。

默认只分析团队级趋势，不展示个人级明细。如需个人级分析（比如辅助某个员工成长），必须由当事人本人授权开启，系统不做默认采集。

用途边界

明确不用于个人绩效评估。能力地图关注的是团队整体的能力分布和协作模式，不生成单人评分、不对个人输出排名、不作为晋升或奖金依据。仅用于系统在给建议时判断"方案是否超出团队能力边界、是否有落地可行性"，目标是让建议更贴近团队实际，而不是评判个人。

1.2 工作内容与项目状态 版本一

框架设计

接入禅道任务流转记录（需求变更、验收标准）、版本发布历史（功能上线记录、延期原因、被砍功能）、复盘文档（经验总结）。系统理解当前在推进的项目、各项目进度状态、历史已完成项目的经验记录、工作流程规范和协作方式。

数据价值

工作内容数据是系统判断"当前团队在做什么、进度如何、是否按预期推进"的依据。历史项目记录则是提炼"什么样的方案容易延期、什么情况下功能会被砍"等规律的原始素材。

1.3 产品现状知识库 版本一

框架设计

整理运营中的产品清单，将各产品的功能架构、核心玩法逻辑、设计意图、版本迭代历史、已知问题和待解决事项整理为结构化文档入库。系统需要理解每个功能存在的原因和它解决的问题，而不只是功能描述本身。

为什么要理解设计意图

只知道功能是什么，系统给出的建议只能是功能层面的改进。理解了"这个功能为什么这样设计"，系统才能判断新方案是否违背原有设计原则、是否会产生冲突、以及是否可以组合使用。

1.4 产品数据基准档案 版本一

框架设计

建立核心业务指标的历史基准档案，包括各指标的正常范围、历史趋势曲线、以及每次版本上线前后的数据变化记录。覆盖六大模块：收益健康、用户留存、付费转化、游戏行为、渠道代理质量、活动效果评估。

双重价值

一方面，基准档案是异常检测的依据——当某指标偏离正常范围时，系统能立即识别并归因。另一方面，这份档案是版本二数字孪生的校准基础——历史上每次版本上线前后的数据变化，就是仿真模型的"标准答案"。

实现方式

通过数数科技 Open API 读取历史数据，按六大模块分类建档。版本一阶段只做基准档案，不做实时监控（实时监控属于版本二）。

1.5 思维方式提炼与隐私保护 版本一

思维方式提炼

通过接入 Lark 工作文档和群聊记录，系统分析真实工作中的问题解决过程、方案讨论路径、决策收敛方式，自动生成思维方式认知文档初稿。管理层审阅确认后写入思想层，与行业层和社会层一起构成系统完整的判断依据。

隐私保护机制

过滤规则由系统执行，不开放给管理员配置。基于人权和隐私原则，只处理工作相关内容，自动过滤个人生活话题、薪资绩效讨论、情绪性吐槽、个人隐私信息。对全体员工透明公示，员工知情。

待讨论分歧点

六个维度的信息接入有优先级差异，建议按价值高低排序推进：产品功能文档 + 数据基准档案（最高优先）→ 复盘文档 + 版本发布记录 → Lark工作文档 → 禅道任务记录 → 代码仓库 → 群聊记录。各数据源的 API 接入成本和可行性需要技术团队评估后确认排期。

版本二模块：决策辅助 + 策略预判

2.1 核心指标监控体系 版本二

框架设计

基于数数科技平台，构建六个维度的目标驱动指标体系：收益健康（充值总额、成功率、ARPU）、用户留存（分群留存曲线、流失预警）、付费转化（注册到首充漏斗、游戏场景触发充值）、游戏行为（游戏切换路径、各游戏用户价值）、渠道代理质量（各渠道LTV对比）、活动效果（ROI、前后留存对比）。

实现原理

通过数数科技 Open API 读取数据，AI 定期扫描各项指标，主动识别异常波动并给出归因分析。不需要人工去查，系统主动推送预警。

2.2 策略效果预判引擎（数字孪生机制） 版本二

框架设计

构建虚拟的用户群和游戏环境。用户画像分为三层叠加：行为层（下注/产出/充值行为）、来源层（注册渠道/代理树）、偏好层（游戏类型偏好：刺激度、交互强度、开奖频率）。在虚拟环境中对新的运营策略进行模拟，预判不同分群用户的响应概率和整体数据变化。

实现原理

运营人员输入新活动的产品文档，AI 读取文档理解策略意图，调取历史规律知识库（类似策略的历史效果），结合当前用户结构，输出分群预测报告。报告包含：有没有效果、效果多大（含置信区间）、有没有副作用。

厂商游戏模块的角色

不仿真游戏内部数值（RTP、赔率由厂商锁定），但将游戏场景状态作为重要变量——玩家当前在玩什么、最近的输赢状态、游戏切换行为，这些直接影响对平台策略的响应概率。同一个充值活动，玩家刚连输三局 vs 刚赢了一把，响应率完全不同。

待讨论分歧点

数字孪生的精度边界：第一版仿真误差预计在 15~30% 范围，需要用历史版本节点数据做回测校准。版本节点即天然的"准A/B数据"——每次版本上线前后的用户行为变化，可作为校准标准答案。校准需要至少 2~3 个历史版本的完整数据对照。

版本三模块：行业视野与竞品洞察

3.1 竞品追踪与行业对标 版本三

框架设计

持续追踪重点竞品（5~10个，名单待补充）的版本更新记录、功能变化、用户评价、广告投放策略。通过时间序列分析，推断竞品在每个阶段面临什么问题、做了什么决策、效果如何。

核心能力

版本考古——将竞品的版本更新记录和公开数据结合，推断"他们在某个阶段做了 X 功能，大概率是为了解决 Y 问题，因为之后他们的 Z 指标出现了明显变化"。阶段对标——分析产品前中后期各阶段的重点方向演变规律，帮助我们判断自己现在处于什么阶段、下一步应该重点做什么。

数据来源说明

只使用公开信息（应用商店、官网、新闻资讯、用户评价、广告素材）。竞品私有运营数据无法获取，通过公开行为推断策略逻辑，这是合法且有效的方式。

待讨论分歧点

竞品名单尚未确认，需要业务团队提供重点追踪对象。跨类型产品的参考范围如何定义——是只看同类棋牌产品，还是扩展到其他实钱游戏品类，需要明确边界。

版本四模块：群体智慧积累与赋能

4.1 职业智慧知识库与多人格协作 版本四

核心能力

系统从员工的历史工作产出中提取正向经验（方案考虑更全面、代码效率更高，且被他人采纳验证）沉淀为职业知识库。员工主动提交文档或代码时，系统基于积累的集体智慧进行审查，补充个人视角可能遗漏的角度，用群体经验提升个体水平。员工离职后经验留存，成为公司的集体资产。

群体智慧的组织方式

系统调用不同职业背景的经验独立思考同一问题，然后呈现多个角度的建议供使用者参考。举例：一份活动方案审查，系统会同时从产品视角（用户体验）、数据视角（转化预期）、风控视角（合规风险）给出反馈——这些视角在单一 AI 里会被折中掉，但在分角色思考下会被完整保留。具体的投票与排序机制属于 V4 启动前再细化的实现细节，会基于前三个版本积累的数据量来决定。

员工成长机制

透明模式，员工自由选择是否开启。系统识别个人在历史工作中的疏忽规律，在相关任务节点针对性提醒。适用条件由系统自动推断（考虑时间节点、资源约束等参数），随使用持续修正。

待讨论分歧点

版本四依赖前三个版本积累足够的数据量才能真正发挥作用，因此启动时间取决于前三个版本的运转周期和数据质量。建议在版本二版本三稳定运行 3~6 个月后，再正式启动版本四的建设。职业知识库的分类粒度（按职能/按项目类型/按决策类型）需要根据实际积累的数据量动态调整。

4.2 专家权重体系与防平庸机制 版本四

核心问题

群体投票容易产生平庸陷阱——多数人倾向选择执行成本低、风险低的普通方案，而专业判断往往来自少数人。如果权重一样，专家的正确判断会被多数人的平庸选择淹没，知识库的质量会随时间退化。

三层权重架构

普通成员（权重 ×1） — 标准投票权重。价值在于提供量和覆盖更多场景视角，但不主导知识库方向。

认证专家（权重 ×3~5，可调） — 由超级账号指定，基于其在特定领域的历史准确率和成果认定。注意：专家资格是领域绑定的，产品策划专家只在产品方案上有高权重，不影响技术架构判断，防止权威外溢。

超级账号（最高权重 + 管理权限） — 可指定和撤销专家资格；可标记某条经验为"强制入库"或"强制排除"；可在系统给出结论时直接覆盖并附注理由；所有覆盖操作自动记录，形成可审计的决策日志。

场景归因分析——防止经验失真

当多数人选择了一个普通方案时，系统不只记录"此方案票数多"，还要分析并记录他们为什么这么选——是执行成本约束、资源限制、还是对更好方案不了解。每条经验在知识库里携带完整的场景条件标签，后续调用时系统会判断当前场景是否匹配，而不是直接套用。

少数专家意见保护机制

当专家判断与多数人不一致时，系统不用票数压制专家意见，而是同时保留两个结论：「多数方案」（适用于资源受限、执行优先的场景）和「专家方案」（适用于条件充足、追求最优的场景），并标注各自适用条件。使用者根据当前实际情况自行判断采用哪个。

打分机制——客观结果优先于主观评价

不依赖人工主观打分，依赖客观结果回流：被采纳的方案上线后数据是否符合预期；被参考的代码有没有出现 bug 或被推翻重写；系统推荐的策略，真实效果和预测是否吻合。结果越好，对应经验的可信度评分越高。时间越长，评分越有根据。

专家资格动态管理

专家资格不是永久的，基于历史准确率动态调整。专家推荐方案结果好则权重维持或提升；连续误判则触发人工复核提醒。系统同时识别领域漂移——专家开始对非擅长领域发表意见时，自动降低其在该领域的权重，防止权威扩散导致判断失真。

防投毒机制

多重验证 — 单一来源的经验不直接入库，需要跨来源、跨时间、跨项目的交叉印证后才提升权重。

来源可信度分级 — 有代码提交记录支撑的 > 有数据结果印证的 > 有文档记录的 > 口头描述的。可信度越低的来源，入库门槛越高。

异常行为检测 — 某个来源在短时间内输入大量与历史模式不符的内容，系统标记为异常，暂缓入库，推送给超级账号人工审核。

冷启动保护期 — 系统上线初期，所有新输入经验进入观察期，只作为参考建议不作为主要权重，直到有足够结果验证后才正式提升权重。

待确认事项

专家初始名单由谁来定、覆盖哪些领域、初始设定几位——这个需要在立项时明确，不能等系统上线后再讨论，否则版本四上线时知识库权重体系无法正常运转。

专家权重倍数（×3 还是 ×5）建议在系统运行 1~2 个月后，根据实际投票分布情况动态校准，不宜在上线前就锁定固定值。

06

风险与挑战

这是一个长期建设项目，会遇到几类风险，需要提前识别和应对。

R1 冷启动期数据不足，早期建议质量有限

风险描述

版本一上线初期，系统接入的历史数据有限，提炼出的认知和建议质量会明显低于中后期。如果管理层和员工在冷启动期对系统抱有过高期望，会产生"不如预期"的失望感，影响后续推进。

应对方式

冷启动期（前 1~3 个月）明确定位为数据积累阶段，系统输出作为参考而非决策依据。人工筛选 20~30 个精华案例作为初始认知底色，让系统有一个合理的起点。所有输出都带置信度标签——高置信度的结论可以信赖，低置信度的明确标注"数据不足"。

R2 AI 判断失误，错误建议被采纳

风险描述

系统会犯错，这是不可避免的。真正的风险是错误的建议被直接采纳并带来业务损失，导致团队对系统彻底失去信任。

应对方式

AI 始终是建议者而非决策者，最终采纳与否由人决定。每个建议附带数据依据和置信度，便于人工判断。系统记录每次建议的采纳结果，持续校准预测准确率。高置信度错误会触发复盘机制，修正知识库。

R3 员工对系统的接受度不足

风险描述

系统会读取员工的日常工作产出（文档、代码、群聊），提取经验。如果员工感到"被监视"或担心自己的工作被 AI 评判，可能产生抵触情绪，甚至故意绕开系统工作，导致数据质量下降。

应对方式

隐私过滤机制由系统强制执行，管理员也无法绕过。员工成长提醒设计为"可选开启"，员工自由决定是否接受 AI 提醒。系统定位强调"提取可复用经验、赋能个体"，不做绩效评估用途。上线前与员工充分沟通系统的工作范围和边界。

R4 思想层定义迟迟不到位

风险描述

思想层需要管理层投入时间正式定义。如果这项工作被其他优先级挤压，系统在面对模糊决策时可能出现判断漂移，无法严格按照公司价值观输出建议。

应对方式

立项批准时同步约定思想层的输入截止时间。可以分阶段输入——先给出粗略的 1.0 版本让系统启动，后续迭代细化。避免追求完美导致无限拖延。

R5 外部环境变化（监管、业务转型）

风险描述

系统基于"印度棋牌平台"这个业务场景建设。如果印度市场监管政策变化，或公司业务方向调整（拓展到其他品类/市场），原有的认知底座和规律库可能需要大量重建。

应对方式

系统架构设计时保持认知底座和业务数据的解耦。思想层和工作方式层可以跨业务复用，产品数据和竞品情报可以按业务线划分。新业务启动时增量扩展，不需要推翻重建。

R6 数据权限与系统审批延迟

风险描述

版本一依赖多个系统的权限接入：Lark 企业自建应用（审批周期 1~2 周）、禅道 API token（需 IT 配合开通）、代码仓库访问权限（需仓库管理员授权）、数数科技 Open API（需获取账号权限）。任何一个环节卡住，都会直接影响启动节奏——开发人员无事可做，项目进度拖延。

应对方式

立项批准当天即启动权限申请流程，不等开发阶段再处理。由项目负责人统一跟进各系统的审批进度，每周同步状态给管理层。同时准备降级方案——如果某个数据源短期无法接入，优先用其他数据源先跑起来，待权限到位后再接入。

异常处理方案

异常类型处理方式告警
API 请求失败（网络/限流）指数退避重试，最多 3 次；失败后写入 sync_error 日志n8n 推送告警到指定频道
数据重复以 source + source_id 做唯一索引，重复时更新而非插入无需告警
内容超长（单条 > 10万字）自动切分后分批处理记录日志
Embedding API 失败放入重试队列，下次定时任务补跑连续失败超 10 条时告警

文档入库优先级

批次内容类型原因
第一批产品功能文档、复盘文档、版本发布记录信息密度最高，是系统理解公司产品和历史决策的核心来源
第二批Lark 工作文档（方案、需求、技术设计）覆盖日常工作讨论，补充第一批的细节
第三批历史会议纪要（文字版）、培训材料有则补充，优先级较低

切分策略

不按文件切分，按语义段落切分（每段 300~500 token）。切分边界优先在标题、空行、句号处断开。每个向量块保留元数据：文档标题、来源、创建时间、作者、所属项目、在原文档中的顺序位置（用于检索后按顺序拼接上下文）。

更新与版本管理

场景处理方式
文档新增切分 → 向量化 → 入库，同时建立文档索引记录
文档修改对比 updated_at 变化，删除旧向量块，重新切分入库；保留历史版本记录用于审计
文档删除软删除：标记 deleted_at，向量块不参与检索但保留原文
来源系统重复内容以文档来源 + 文档 ID 做唯一键，避免同一文档被重复入库

人天估算依据

切分逻辑 1~2 天，Embedding API 集成 1 天，pgvector 表结构 + 索引 1 天，批量历史文档入库脚本 1~2 天，检索接口封装 1~2 天。合计 5~8 天。

第三层：AI 语义分类

对规则层无法明确判断的内容（置信度低），调用轻量 AI 模型（Claude Haiku，成本约 $0.001/千 token）做二次判断。分类结果：工作内容（通过）/ 个人私事（拦截）/ 不确定（待审核）。待审核内容推送给系统管理员处理，处理结果反馈给规则库持续优化。

待确认

版本一整体架构小结：n8n 负责数据调度 → 隐私过滤中间件拦截敏感内容 → Supabase（PostgreSQL + pgvector）统一存储原始数据和向量 → Claude API 做内容提炼和认知文档生成 → Retool 提供管理界面。所有组件均可部署在同一台服务器，运维复杂度低，适合小团队维护。
